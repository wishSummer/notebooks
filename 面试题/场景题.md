# 应用场景题

### 缓存穿透

- 场景：
    查询 key 值不存在。例如 用户恶意仿造不存在的 key 发起请求，大量请求访问数据库，而数据库没有匹配数据，也无法放入缓存，导致消耗大量数据库资源。

- 解决方案
  - key 值合法性校验，符合规则进行查询
  - 看使用场景是否适合使用布隆过滤器
    - 底层使用一个 bit 数组，通过对 key 进行多次 hash 计算会得到多个位置，通过将 bit 数组对应位置值设置为 1 表示存在值。
    - 缺点 1 ：hash 结果存在重复情况
    - 缺点 2 ：当数据库更新后， bit 数组实时同步更新问题
  - 缓存空值 key
  - API 接口防护
    - 对接口根据 ip 、 session等信息进行访问频率限制。

### 缓存击穿

- 场景：
    当某个热点缓存数据时效时，如果大量用户请求同一个热点数据，会造成数据库瞬间压力过大，导致直接挂掉。

- 解决方案：
  - 使用互斥锁，确保相同请求，只有一个请求访问到数据库并更新缓存，其他请求则等待
  - 将异步任务更新缓存数据过期时间
  - 将key设置为··永不过期··，后期可以手动进行删除


### 缓存雪崩

- 场景：
    多个热门 key 同时失效，或是缓存服务down机。

- 解决方案：
  - 缓存 key 到期时间 + 随机数，避免同时失效多个 key
  - 若 缓存服务发生故障，可以采用服务熔断或请求限流来应急，后续服务恢复后再对热点 key 进行预热
  - 针对缓存服务器采用一些高可用框架：哨兵模式、集群模式等。避免单个节点出问题导致服务down机
    - 使用哨兵模式，当某个master服务下线，自动将该master下的某个slave服务升级为master服务，继续处理请求
    - 服务降级，若多个请求在短时间内都获取数据失败，则开启全局开关，后续请求直接读取预设默认值
